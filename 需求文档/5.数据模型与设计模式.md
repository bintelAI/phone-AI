## 一、核心数据模型

### 1.1 Agent 动作模型

```kotlin
data class ParsedAgentAction(
    val metadata: String,           // "do" 或 "finish"
    val actionName: String?,        // 动作名称（tap, swipe, type, back, home, wait, launch 等）

data class AgentResult(
    val success: Boolean,           // 是否成功
    val message: String,            // 结果消息
    val steps: Int,                 // 执行步数
)
```


## 二、设计模式应用

### 2.1 门面模式 (Facade)

**VirtualDisplayController** 作为虚拟屏系统的统一门面：
```kotlin
object VirtualDisplayController {
    fun prepareForTask(context, adbExecPath): Int?
    fun screenshotPngBase64NonBlack(): String?
    fun injectTapBestEffort(displayId, x, y): Boolean
    fun cleanup(context)
}
```

将 ShizukuVirtualDisplayEngine、VdGlFrameDispatcher、InputHelper 等复杂子系统隐藏在门面之后。

### 2.2 单例模式 (Singleton)

**全局状态管理**：
```kotlin
object AppState {
    private var appContext: Context? = null
    fun getAppContext(): Context? = appContext
}

object VirtualDisplayController  // 虚拟屏控制器

object AutomationOverlay          // 自动化覆盖层
```

### 2.3 策略模式 (Strategy)

**截图优化策略**：
- ScreenshotCache：缓存策略
- ScreenshotThrottler：节流策略
- ScreenshotOverlayGuard：悬浮窗防护策略

**模型调用策略**：
- AutoGlmClient：云端模型调用
- LocalMnnInferenceEngine：本地模型推理
- OpenAICompatibleProvider：第三方 API

### 2.4 仓储模式 (Repository)

```kotlin
// 数据访问的统一抽象
class ConversationStorageRepository(private val dao: ConversationDao)
class AppPreferencesRepository(private val dataStore: DataStore<Preferences>)
class ToolPermissionsRepository  // 工具权限管理
class VirtualDisplayConfigRepository  // 虚拟屏配置
```

### 2.5 依赖注入模式 (DI - Koin)

**模块化注入**：
```kotlin
startKoin {
    modules(appModule, dataModule, networkModule, uiModule)
}
```

- **AppModule**：全局单例（TelemetryHeartbeatManager、ImageLoader）
- **DataModule**：数据库、仓储
- **NetworkModule**：OkHttp 客户端、AI Provider
- **UiModule**：ViewModel 等 UI 层依赖

### 2.6 MVVM 架构模式

```
View (Composable) ←→ ViewModel ←→ Repository ←→ Data Source
```

- **View**：Composable 函数（HomeScreen、SettingsScreen 等）
- **ViewModel**：管理 UI 状态和业务逻辑
- **Repository**：统一数据访问接口
- **Data Source**：Room DB、DataStore、网络 API

### 2.7 观察者模式

- **StateFlow**：ViewModel 通过 StateFlow 暴露状态给 UI
- **LiveData**：部分模块使用 LiveData 观察数据变化
- **Flow**：Room 数据库查询返回 Flow，自动响应数据变化

### 2.8 工厂模式

- **AgentConfiguration.DEFAULT** 和 **AgentConfiguration.TEST**：预定义配置工厂
- **AIProvider**：不同 AI 服务提供商的工厂创建

### 2.9 适配器模式

- **OpenAICompatibleProvider**：将不同模型 API 适配为统一接口
- **ShizukuBridge**：将 Shizuku API 适配为应用内调用

### 2.10 责任链模式

- **自动化动作执行链路**：解析 → 修复 → 执行 → 重试 → 降级
- **权限检查链路**：敏感词检测 → 二次确认 → 执行/拦截

## 四、错误处理策略

### 4.1 模型调用失败
- 自动重试（最多 3 次，指数退避）
- 网络错误重试、非关键错误重试
- 关键错误直接返回

### 4.2 动作解析失败
- 修复提示（让模型重新输出）
- 最多 2 次解析修复
- 修复失败则任务终止

### 4.3 动作执行失败
- 截图并让模型重新规划
- 最多 1 次动作修复
- 修复失败则任务终止

### 4.4 虚拟屏异常
- 聚焦丢失 → 100ms 高频恢复
- 截图黑帧 → 自动等待非黑帧
- IME 死锁 → 专用控制器防护
